# Article Digest Grabber

Набор скриптов для сбора, обработки и импорта новостных статей в Notion.

## Рабочий процесс

```
parse_articles.py  →  find_duplicates.py  →  find_notion_duplicates.py  →  process_with_deepseek.py  →  collect_tags.py  →  import_to_notion.py  →  create_digest.py
    парсинг          поиск дубликатов       дубликаты vs Notion              переработка заголовков        теги Notion             импорт           создание дайджеста
```

---

## Установка

```bash
pip install -r requirements.txt
```

> Для `find_duplicates.py` дополнительно нужна модель LaBSE (~1.8 GB) — скачивается автоматически при первом запуске и кешируется в `~/.cache/huggingface/hub/`.

## Настройка переменных окружения

Создайте файл `.env` в корне проекта:

```
NOTION_API_KEY=secret_xxx...
DEEPSEEK_API_KEY=sk-xxx...
```

Все скрипты читают ключи из этого файла автоматически. Передавать ключи аргументами командной строки больше не нужно.

> `.env` уже добавлен в `.gitignore` — ключи не попадут в репозиторий.

## 1. Парсинг статей — `parse_articles.py`

Извлекает заголовок, дату публикации, текст и изображения из статей.

### Поддерживаемые сайты

| Сайт | Описание |
|---|---|
| `habr.com` | Российский IT-портал |
| `vc.ru` | Российский бизнес-портал |
| `techcrunch.com` | Технологический новостной сайт |
| `news.crunchbase.com` | Новости стартапов |
| `infoq.com` | IT-портал (без изображений — base64) |
| Другие | Универсальный парсинг |

### Особенности

- Параллельные запросы к нескольким URL одновременно
- Retry-логика при ошибке 403 (до 3 попыток с нарастающей задержкой)
- Случайные задержки 0.1–0.5 с между запросами для имитации браузера

### Использование

```bash
# Один URL
python3 parse_articles.py "https://habr.com/ru/news/123456/"

# Несколько URL
python3 parse_articles.py "https://vc.ru/media/123?from=rss" "https://techcrunch.com/2025/11/09/article/"

# Из файла (рекомендуется)
python3 parse_articles.py --file my_urls.txt
```

> **Важно:** URL с параметрами (`?`, `&` и т.д.) обязательно заключайте в кавычки.

Формат `my_urls.txt` — один URL на строку:

```
https://habr.com/ru/news/123456/
https://techcrunch.com/2025/11/09/article/
https://vc.ru/media/123456
```

### Результат

**`parsed_articles.json`** — все статьи в JSON:

```json
[
  {
    "url": "https://vc.ru/media/123456",
    "site_type": "vcru",
    "title": "Заголовок статьи",
    "date": "10.11.2025",
    "text": "Полный текст...",
    "images": [{"url": "https://example.com/image.jpg", "alt": "Описание"}],
    "status": "success"
  }
]
```

**`articles_markdown/NNN_Заголовок.md`** — отдельный файл на каждую статью:

```markdown
# Заголовок статьи

**Дата публикации:** 10.11.2025
**Источник:** https://vc.ru/media/123456

---

Текст статьи...

![Описание](https://example.com/image.jpg)
```

### Решение проблем

**`zsh: no matches found`** — заключите URL в кавычки:
```bash
# Неправильно:
python3 parse_articles.py https://example.com/article?id=123

# Правильно:
python3 parse_articles.py "https://example.com/article?id=123"
```

## 2. Поиск дубликатов — `1.5_find_duplicates.py`

Автоматически находит статьи об одном и том же событии из разных источников.

### Алгоритм (гибридный)

1. **Двойной сигнал LaBSE** — вычисляются два независимых косинусных сходства для каждой пары: по полному тексту и только по заголовкам. Пара становится кандидатом если хотя бы один сигнал превышает порог. Это позволяет находить RU↔EN дубликаты и статьи написанные с разных углов об одном событии.
2. **DeepSeek Chat (chain-of-thought)** — проверяет кандидатные пары по шагам: определяет субъект и событие каждой статьи, сравнивает. В промпт передаются даты публикации — совпадение дат усиливает сигнал. Пары с очень высоким сходством (≥ 0.85) подтверждаются автоматически без API-запроса.
3. **Транзитивная группировка** — если A=B и B=C, все три объединяются в одну группу в отчёте. Из группы остаётся английская статья (язык определяется по телу статьи, не по заголовку — заголовки уже переведены на русский скриптом `2_new_title_with_deepseek.py`). Если все статьи на одном языке — остаётся первая по алфавиту.

### Использование

```bash
# Только отчёт (файлы не трогаются)
python3 1.5_find_duplicates.py

# Переместить дубликаты в articles_duplicates/
python3 1.5_find_duplicates.py --action move
```

### Параметры

| Параметр | По умолчанию | Описание |
|---|---|---|
| `--threshold` | `0.58` | Нижний порог сходства по тексту для LLM-проверки |
| `--title-threshold` | `0.52` | Нижний порог сходства по заголовку для LLM-проверки |
| `--confirm-threshold` | `0.85` | Выше этого — дубликат без LLM |
| `--action` | `report` | `report` — только отчёт, `move` — переместить дубликаты |
| `--concurrent` | `5` | Параллельных запросов к DeepSeek |

### Результат

**`duplicates_report.txt`** — отчёт сгруппированный по событиям. Пара из двух статей и группа из трёх выглядят одинаково:

```
#1  (2 статьи об одном событии)
  ОСТАВИТЬ:  023_OpenAI_releases_GPT-5.md  [EN]
             OpenAI releases GPT-5 to everyone
  УДАЛИТЬ:   001_OpenAI_выпустила_GPT-5.md  [RU]  (сходство: 91.3%)
             OpenAI выпустила GPT-5
             Причина: Одно событие, русская версия EN статьи

#2  (3 статьи об одном событии)
  ОСТАВИТЬ:  045_Meta_launches_Llama-4.md  [EN]
             Meta launches Llama-4
  УДАЛИТЬ:   031_Мета_выпустила_Llama-4.md  [RU]  (сходство: 93.1%)
             Мета выпустила Llama-4
             Причина: Одно событие, разные источники
  УДАЛИТЬ:   067_Llama-4_новая_модель_Meta.md  [RU]  (сходство: 88.5%)
             Llama-4 — новая модель от Meta
             Причина: транзитивный дубликат

ФАЙЛЫ ДЛЯ УДАЛЕНИЯ:
  - 001_OpenAI_выпустила_GPT-5.md
  - 031_Мета_выпустила_Llama-4.md
  - 067_Llama-4_новая_модель_Meta.md
```

При `--action move` дубликаты перемещаются в `articles_duplicates/` (можно восстановить вручную).

## 2.5. Проверка дубликатов в Notion — `1.6_find_notion_duplicates.py`

Сравнивает локальные статьи из `articles_markdown/` со статьями, уже загруженными в выбранную базу данных Notion. Позволяет не импортировать то, что в дайджесте уже было.

### Алгоритм

1. **Двойной сигнал LaBSE** — вычисляются эмбеддинги полного текста и отдельно заголовков для локальных статей и страниц Notion. Кросс-матрицы сходства N×M. Пара становится кандидатом если хотя бы один сигнал превышает порог (текст ≥ 0.58 или заголовок ≥ 0.52).
2. **Авто-дубликаты** — пары с сходством по тексту ≥ 0.85 подтверждаются без LLM.
3. **DeepSeek-верификация (chain-of-thought)** — кандидаты проверяются с передачей дат публикации и пошаговым рассуждением (опционально).

### Использование

```bash
python3 1.6_find_notion_duplicates.py
```

Скрипт интерактивно запросит:
1. Notion Integration Token (или `NOTION_API_KEY` из окружения)
2. Базу данных Notion — выбор из списка или ввод URL/ID вручную
3. Период: последние 7 / 30 / 90 / 365 дней, весь период или произвольный диапазон (формат дат: `DD-MM-YYYY`)
4. DeepSeek API ключ (или `DEEPSEEK_API_KEY`) — можно пропустить, тогда только LaBSE
5. Пороги сходства (Enter для дефолтов: 0.58 / 0.85)
6. Загружать ли содержимое блоков страниц Notion (медленнее, точнее — актуально если текст статей хранится внутри Notion, а не как внешняя ссылка)

> Фильтрация страниц Notion выполняется по свойству **«Дата публикации»** (тип Date) — тому же полю, которое заполняет `4_import_to_notion.py`. Системное поле `created_time` не используется.

### Результат

**`notion_duplicates_report.txt`** — список локальных файлов с совпадениями:

```
#1  ЛОКАЛЬНАЯ СТАТЬЯ [RU]:
     060_Mistral_запустила_Forge.md
     Mistral запустила Forge — платформу для обучения корпоративных LLM с нуля

     Совпадения в Notion (1):
       — [EN] Mistral launches Forge for enterprise LLM training
         Создана: 2026-03-01  |  Сходство: 91.2%
         URL: https://notion.so/...
         Причина: LaBSE сходство 91.2% (авто)

ФАЙЛЫ-ДУБЛИКАТЫ (рекомендуется не включать в дайджест):
  - 060_Mistral_запустила_Forge.md
```

Файлы при этом **не перемещаются и не удаляются** — только отчёт.

## 3. Переработка заголовков — `process_with_deepseek.py`

Отправляет текст каждой статьи из `articles_markdown/` в DeepSeek API и заменяет H1-заголовок (`# ...`) на ответ модели.

### Использование

```bash
python3 2_new_title_with_deepseek.py
```

Скрипт интерактивно запросит:
1. Файл системного промпта (любой `.txt` файл в корне проекта)
2. Количество параллельных запросов (по умолчанию 5)

### Логика работы

1. Извлекает тело статьи (всё после `---`)
2. Удаляет изображения из текста
3. Отправляет очищенный текст в DeepSeek API с выбранным системным промптом
4. Заменяет H1-заголовок на ответ API
5. Выводит итоговый отчёт со старыми и новыми заголовками

### Файл системного промпта

Любой `.txt` файл — его содержимое передаётся в параметр `system` при каждом запросе. Определяет задачу для модели (генерация заголовков, суммаризация и т.д.).

## 4. Сбор тегов — `collect_tags.py`

Выполняет две задачи:
1. Собирает теги из Multi-Select поля Notion Database → сохраняет в `tags.txt`
2. Ищет упоминания тегов в `.md` файлах и проставляет блок `**Компания:**` с найденными значениями

> Название блока `**Компания:**` захардкожено в скрипте — вне зависимости от того, как называется поле в Notion, из которого собирались теги.

### Настройка Notion API

1. Перейдите на [notion.so/my-integrations](https://www.notion.so/my-integrations)
2. Создайте интеграцию и скопируйте "Internal Integration Token"
3. В Notion Database: `...` → `Connections` → `Add connections` → выберите интеграцию

### Использование

```bash
python3 3_collect_tags.py
```

### Как работает сбор тегов

1. Запрашивает URL или ID Database
2. Выводит все поля Database; поля `multi_select` отмечены символом `✓`
3. Пользователь выбирает поле (если оно одно — берётся автоматически)
4. Выгружает все страницы с пагинацией, извлекает уникальные значения
5. Сохраняет в `tags.txt` (один тег на строку, по алфавиту)

Если `tags.txt` уже существует, скрипт предложит использовать его или пересобрать.

### Как работает автозаполнение тегов

После сбора тегов скрипт предлагает проставить их в `.md` файлы. Интерактивно запрашивает:

**Нормализация** — убирать ли знак в конце тега при поиске:
- `Apple.` → ищет `Apple` в тексте, но записывает `Apple.` в файл (сохраняет формат Notion)

**Область поиска:**

| Вариант | Где ищется |
|---|---|
| 1 | Заголовок + всё тело статьи |
| 2 | Заголовок + первый абзац тела |
| 3 | Только заголовок |

**Регистр:** строгое соответствие или без учёта (по умолчанию — без учёта).

Файлы, в которых блок `**Компания:**` уже есть, пропускаются.

### Формат блока в .md файле

```markdown
# Название статьи
**Дата публикации:** 02.02.2026
**Источник:** https://example.com/article
**Компания:** Apple., Google., Notion.

---

Тело статьи...
```

### Устранение проблем

| Ошибка | Причина и решение |
|---|---|
| `Unauthorized` | Неверный токен или интеграция не подключена к Database |
| `Object not found` | Неверный Database ID или URL |
| Теги не находятся | Включите нормализацию, если теги в Notion заканчиваются на `.` |
| Файл пропускается | Блок `**Компания:**` уже есть — удалите его вручную для повторной обработки |

## 5. Импорт в Notion — `import_to_notion.py`

Импортирует статьи из `articles_markdown/` в Notion Database.

### Настройка Notion API

Те же шаги, что и для `collect_tags.py` (см. раздел выше).

**Database ID** извлекается из URL:
```
https://www.notion.so/workspace/abc123def456ghi789jkl012mno345pq?v=...
                                 ^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^^
                                 это и есть Database ID
```

### Структура Database

Обязательные поля:

| Поле | Тип |
|---|---|
| `Name` | Title |
| `URL` | URL |
| `Дата публикации` | Date |

Поля типа `Multi_select` (например, "Компания") подхватываются автоматически.

### Использование

```bash
python3 4_import_to_notion.py
```

### Что импортируется

**Properties:**
- `Name` ← заголовок статьи
- `URL` ← URL статьи
- `Дата публикации` ← дата (DD.MM.YYYY)
- Multi-select поля ← если есть в Database и выбраны при запуске

**Контент страницы:** только тело статьи (текст, изображения, списки). Заголовок, URL и дата в контент не добавляются — они в properties.

> Если в `.md` файле более 100 блоков — Notion API не позволяет добавить их за один запрос. Скрипт обходит это ограничение, добавляя блоки после создания страницы.

### Устранение проблем

| Ошибка | Причина и решение |
|---|---|
| `Unauthorized` | Неверный токен или интеграция не подключена к Database |
| `Object not found` | Неверный Database ID |
| Properties не заполняются | Проверьте точность названий и типов полей в Database |
| Multi-select не импортируется | Убедитесь, что значения в `.md` разделены запятыми и вы выбрали поле при запуске |

## 6. Создание дайджеста — `create_digest.py`

Финальный этап. Создаёт страницу-дайджест в Notion Database "Личный блог", собирает новости из Database "Обзор рынка технологии машинного обучения" за указанный период и заполняет страницу по шаблону.

### Использование

```bash
python3 5_create_digest.py
```

Скрипт интерактивно запросит:
1. URL или ID Database "Личный блог" (куда создаётся страница)
2. URL или ID Database с новостями
3. Путь к файлу шаблона
4. Диапазон дат для сбора новостей

### Что делает скрипт

1. Создаёт страницу `AI Digest - Week N YEAR` в Database "Личный блог" по шаблону (секции Research, Notes, Draft)
2. Собирает новости из Database за указанный период — фильтрация по полю **«Дата публикации»**
3. Проверяет соответствие названий статей фактическим заголовкам на сайтах — результат сохраняется в `logs/title_verification_weekN_YEAR.log`
4. Переводит английские заголовки на русский
5. Агрегирует новости по датам
6. Добавляет новости в секцию Draft страницы

### Файл шаблона

Любой `.md` файл, определяющий структуру страницы дайджеста. По умолчанию используется `my_digest_template.md` (в `.gitignore`). Шаблон содержит секции, которые будут созданы на странице Notion.
