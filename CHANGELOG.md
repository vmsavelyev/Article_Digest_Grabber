# Changelog

## v2.0 — Улучшение детекции дубликатов + единое поле «Дата публикации»

### 1.5_find_duplicates.py · 1.6_find_notion_duplicates.py

#### Новое: двойной сигнал LaBSE

Раньше кандидаты отбирались только по косинусному сходству **полного текста**. Теперь вычисляются два независимых сигнала:

- Сходство **полного текста** (заголовок + тело)
- Сходство **только заголовков**

Пара становится кандидатом если хотя бы один сигнал превышает порог. Это позволяет находить дубликаты в двух сложных случаях:
- Статьи написаны с разных углов (B2B vs B2C) — тело разное, заголовки похожи
- RU↔EN пары — LaBSE кросс-лингвальная, но при длинном тексте сходство размывается

#### Новое: дата публикации в LLM-промпте

LLM теперь получает дату публикации каждой статьи. Если даты совпадают — в промпт добавляется явная подсказка:
> «Обе статьи опубликованы в ОДИН ДЕНЬ (16.04.2026) — это сильный признак одного события.»

Это помогает отличать дубликаты от тематически похожих статей разных дней.

#### Улучшен LLM-промпт: chain-of-thought

Вместо прямого вопроса «это дубликат?» модель теперь рассуждает по шагам:
1. Кто главный субъект статьи 1?
2. Какое событие описано в статье 1?
3. Кто главный субъект статьи 2?
4. Какое событие описано в статье 2?
5. Одно и то же событие?

Промпт явно описывает сложные случаи: кросс-языковые пары, разная терминология, разные углы подачи одного события.

#### Изменены пороги

| Параметр | Было | Стало |
|---|---|---|
| Порог кандидата (текст) | 0.65 | 0.58 |
| Порог авто-дубликата | 0.90 | 0.85 |
| Порог кандидата (заголовок) | — | 0.52 (новый) |

Снижение порога кандидата расширяет воронку — больше пар доходит до LLM-проверки. Снижение авто-порога позволяет подтверждать высококонтентные кросс-языковые пары без LLM-запроса.

#### Увеличен лимит токенов ответа LLM

`max_tokens`: 150 → 400. При 150 токенах модель не успевала написать полные рассуждения по 5 шагам и могла обрезать `reasoning` на полуслове.

#### Новый флаг CLI (1.5)

```bash
--title-threshold 0.52   # порог сходства только по заголовкам
```

#### Улучшен лог

Для каждой проверяемой пары теперь выводятся оба сигнала:
```
[✓] article_a.md ↔ article_b.md  (текст=0.74, заголовок=0.49)
```

### 1.6_find_notion_duplicates.py

#### Фильтрация по «Дата публикации» вместо created_time

Раньше скрипт фильтровал страницы Notion по системному полю `created_time` — дате создания записи в Notion. Это не совпадает с датой публикации статьи, поэтому большинство записей выпадало из выборки. Теперь фильтр применяется к свойству **«Дата публикации»** — тому же полю, которое заполняет `4_import_to_notion.py`.

#### Прямые HTTP-запросы к Notion API через httpx

Заменён нестабильный вызов `notion.data_sources.query()` на прямой POST-запрос к `https://api.notion.com/v1/databases/{id}/query` через `httpx` — по той же схеме, что использует `5_create_digest.py`. Устранена ошибка `'DatabasesEndpoint' object has no attribute 'query'`.

#### Исправлен формат ввода дат

Ручной ввод периода теперь принимает формат `DD-MM-YYYY` вместо `YYYY-MM-DD`.

#### Исправлена ошибка включения последнего дня

При ручном вводе конечная дата ранее интерпретировалась как `00:00:00` этого дня, и все статьи, опубликованные в этот день, выпадали. Теперь используется фильтр `on_or_before`, который корректно включает весь последний день.

#### Исправлен NameError: title_threshold

Переменная `title_threshold` использовалась в цикле сравнения, но нигде не определялась. Заменена на константу `DEFAULT_TITLE_THRESHOLD`.

Логика поиска поля для даты теперь в первую очередь ищет поле с точным именем **«Дата публикации»**, и только если такого нет — ищет любое поле типа `date`.
